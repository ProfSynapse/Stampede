# Stampede

Config-driven k6 load testing tool. Separates **profiles** (what to test) from **patterns** (how to test).

## Quick Start

```bash
# 1. Create a profile from the example template
cp profiles/example.js profiles/myapp.js
# Edit profiles/myapp.js with your target URL, auth, and endpoints

# 2. Activate it as the target
ln -sf myapp.js profiles/target.js

# 3. Run a pattern
k6 run patterns/smoke.js
```

## Structure

```
stampede/
  profiles/         Target configs (URL, auth, endpoints)
    example.js      Template — copy and customize
    _base.js        Fallback defaults
    target.js       Active profile (symlink, gitignored)
  patterns/         Test shapes (spike, stress, breakpoint, etc.)
  lib/              Shared utilities (auth, metrics, HTTP, WebSocket)
  skills/           Canonical agent config (single source of truth)
    agent-instructions.md   Agent instructions (synced to .claude/ and .codex/)
    skill.md                Skill definition
    scripts/                Agent scripts (setup, run, status, teardown)
    templates/              Profile templates
  .claude/          Claude Code config (generated by sync.sh)
  .codex/           Codex config (generated by sync.sh)
  sync.sh           Distributes skills/ to provider directories
  run.sh            CLI entrypoint (subcommands: run, list, help)
  Dockerfile        Alpine + k6 image
  railway.json      Railway one-off worker config
```

## Key Conventions

- **CLI-driven**: `./run.sh run --profile myapp --pattern spike --max-vus 2000`. CLI args for all tunables; env vars only for secrets (`EMAIL`, `PASSWORD`, `HMAC_SECRET`, etc.).
- **profiles/target.js** is what patterns import. Set it via `ln -sf myapp.js profiles/target.js` or via `run.sh run` (which does this automatically from `--profile`).
- **Profiles are gitignored** except `example.js` and `_base.js`. Never commit credentials.
- **Auth strategies**: `form_login`, `hmac`, `api_key`, `bearer`, `none` — configured in the profile.
- **Env var fallbacks**: `STAMPEDE_PROFILE`, `STAMPEDE_PATTERN`, `STAMPEDE_MAX_VUS`, `STAMPEDE_DURATION`, `STAMPEDE_BASE_URL`, `STAMPEDE_AUTH_STRATEGY` — CLI args take precedence.
- Profiles must export `contentPath(template)` as a named export (used by content-heavy patterns).

## Available Patterns

| Pattern | What It Does |
|---------|-------------|
| `smoke` | 2 VUs, 30s sanity check |
| `stress` | Ramping sustained load (configurable max) |
| `spike` | Escalating spikes with recovery |
| `breakpoint` | Scale until it breaks (HTTP + optional WebSocket) |
| `school-bell` | Jagged spikes simulating class schedules |
| `concurrency` | Flat instant-jump stress (no ramp) |

## Running via Docker

```bash
docker build -t stampede -f Dockerfile .
docker run --rm stampede run --profile myapp --pattern smoke
docker run --rm \
  -e EMAIL=user@example.com \
  -e PASSWORD=secret \
  stampede run --profile myapp --pattern spike --max-vus 2000
```

## Agent Skills

The `skills/` directory is the single source of truth for agent configuration. Edit files there, then run `./sync.sh` to distribute to provider directories (`.claude/`, `.codex/`).

```bash
# Edit the canonical source
vim skills/agent-instructions.md

# Sync to all providers
./sync.sh

# Check if providers are in sync (CI-friendly, exits 1 if drifted)
./sync.sh --check
```

AI agents can operate Stampede through the skill. Ask:
- "Set up Stampede for https://my-app.com"
- "Run a spike test with 2000 VUs"
- "Show me the available patterns"
